# Phase P4 — Optimizer + Apply (the headline)

**Goal:** The parameter-sweep optimizer (ComboGenerator/SweepRanker/SweepRunner/Orchestrator on a spawn ProcessPool + shared-memory snapshot) + `optimize_config`/`sweep_*` tools, the baseline+uplift+robustness-verdict logic, the `mcp_proposals` propose→human-apply→revert pipeline, the Proposals + Optimizer UI, and the live-trading-protection gates (live-order p95/p99 under sweep, RSS/loop breaker, reserved DB floor).

**Entry:** P3 exit met (`run_one`, `source/sweep_id`, rate-gate lane green).
**Exit:** golden-sweep acceptance (AC-007 winner==expected hash + full FR-018 bar); apply-sanitization + drift-guard + deny-list call-graph + worker-env-scrub tests green; **live order-placement p95 ≤ 1.15× / p99 ≤ 1.3× baseline during a max sweep** (AC-011); null-result honesty (AC-006); constraint-exclude (AC-024); re-rank (AC-025).

**Requirements:** FR-015..025/039/040, NFR-002/005/008, AC-005..009/020/023/024/025/026; ADR-6/8, §15.1/15.5/15.12/15.14.

## K. Backend Implementation Plan (`backend/mcp/tools/optimizer/`)

### Sweep engine (decomposed)
- TASK-P4-01: `combos.py` `ComboGenerator` — pure. `generate(param_space, strategy: 'grid'|'random', seed) -> list[dict]`: grid = cartesian product (canonical key-sorted, type-normalized); random = seeded N distinct draws; each combo gets `config_hash = sha256(canonical_json(config))`. Dedup by hash. Refuse if `count > MCP_MAX_SWEEP_BACKTESTS` (5000) or empty space (pre-flight). **Hypothesis test:** count==product, zero dupes, in-bounds, cap at exactly 5000, seed-reproducible.
- TASK-P4-02: `ranker.py` `SweepRanker` — pure. `rank(results, objective, constraints) -> ranked`: exclude configs violating constraints (FR-039) BEFORE ranking; sort by `objective_value DESC NULLS LAST`; deterministic tie-break (secondary metric → config_hash); NaN/Inf→None quarantine (sort last). Reuses the existing backtest metrics module — no re-implemented math. Computes the full 8-metric superset per result.
- TASK-P4-03: `snapshot.py` — load the kline snapshot ONCE (gathered fetches, incl. BTC), pack COLUMNAR into `multiprocessing.shared_memory` (off the event loop, in a thread); per-sweep RSS budget pre-flight (shm free-space ≥ budget via posix_fallocate→catchable ENOSPC, never SIGBUS — §15.12). Deterministic `close()+unlink()` in finally; boot unlinks orphans.
- TASK-P4-04: `runner.py` `SweepRunner` + `_worker.py` (module-level, picklable, sync). **CRITICAL worker model (backend-R1-F1/F3/F5):** the async `BacktestService.run_one` is used ONLY for the in-process baseline (TASK-P4-06) — it is a bound method holding a ThreadPoolExecutor/DB pool/Events and is NOT picklable for `spawn`, and a coroutine submitted to a ProcessPool would never be awaited. For workers, ship a **module-level SYNC entrypoint** `def _run_combo(combo_config: dict, shm_name: str, signals: list[dict], instrument_info: dict, deadline_epoch: float) -> dict` in `_worker.py` that: imports `BacktestEngine` FRESH in-worker (it already imports lazily — `backtest_service.py:662,728`), reconstructs row-oriented klines from the attached shm (reslice the columnar buffer per window — backend-R1-F11), builds a `threading.Event` from `deadline_epoch`, calls `engine.run(config, signals, klines, cancel_event, on_progress=None, instrument_info)` SYNCHRONOUSLY, replicates `_attach_buy_hold` from the shm's BTCUSDT, returns a plain metrics dict. `signals`+`instrument_info`+`shm_name` are passed ONCE via `ProcessPoolExecutor(initializer=_worker_init, initargs=(shm_name, signals, instrument_info))` (read-only per-worker), so `submit(_run_combo, combo_config)` carries only the tiny combo config.
  - `ProcessPoolExecutor(max_workers=max(1, min(cores-1, MCP_SWEEP_WORKERS)), mp_context=get_context('spawn'), initializer=_worker_init)`. `_worker_init` SCRUBS secrets from `os.environ` (ACCOUNTS_ENCRYPTION_KEY/DATABASE_URL/MCP_TOKEN/*_API_KEY), sets `os.nice(+10)` + high `oom_score_adj` (POSIX-guarded — no-op on Windows), attaches the shm.
  - **Workers are DB-LESS** (DATABASE_URL scrubbed): a worker RETURNS its metrics dict; the **PARENT-side** `SweepRunner` collects each Future and persists via `SweepRepository` (TASK-P4-12b) in its OWN per-combo committed txn, incrementing `completed_combos`. (backend-R1-F3: do NOT attempt a worker-side DB write.)
  - Cancellation = terminate worker PIDs via managed Process/Future handles (never raw os.kill — security-R4-F6). BrokenProcessPool (oom-kill) → shed/backoff/resume from completed-config-hash set (no on-loop rebuild).
- TASK-P4-05: `orchestrator.py` `SweepOrchestrator` — thin lifecycle: `start_sweep(spec) -> sweep_id` (background task, strong-ref set + done-callback), `status(sweep_id)`, `results(sweep_id, objective=None)` (re-rank stored rows by an alternate metric — FR-040), `cancel(sweep_id)`. Boot-recovery: claim `running`→`interrupted` (atomic `UPDATE … RETURNING`) BEFORE advertising tools; resume by completed-config-hash set.
- TASK-P4-06: baseline — run the CURRENT live `AutoTradeConfig` through `run_one` on the same snapshot to get incumbent metrics; every candidate reports uplift (Δreturn/ΔmaxDD/ΔSharpe/Δexpectancy) vs baseline; harness-locked (identical capital/fees/slippage/range/universe). Robustness verdict per FR-020 (hard/soft checks). Null-result honesty (FR-018): if no candidate clears the bar → "keep current config", no winner.

### Optimizer tools

### Apply / proposal pipeline (`backend/mcp/` control-plane + `repositories/proposal_repo.py`)
- TASK-P4-08: `ProposalRepository` — create (`pending`, target_schedule_id, target_config_index, config patch, diff.before=full prior AutoTradeConfig, config_schema_version, expires_at), get, list, status transitions (validated + audited).
- TASK-P4-09: emit a winning config as a `mcp_proposals` row (FR-022); the agent gets `proposal_id` + deep link; agent CANNOT apply.
- TASK-P4-10: `POST /api/v1/mcp/proposals/{id}/approve` (control-plane, human, app-auth) — the apply pipeline:
  1. load proposal; reject if expired (TTL) or `config_schema_version` stale (coerce-or-reject).
  2. **allow-list sanitize** — accept ONLY the literal `SWEEPABLE_FIELDS` frozenset; reject anything else (fail-closed; a CI test fails if `AutoTradeConfig` gains an unclassified field).
  3. **absolute sanity ceiling** — hard max leverage / min stop distance / max position fraction / max concurrent / TP-SL ordering; reject out-of-band.
  4. reject if `target_schedule_id IS NULL`; bounds-check `0 ≤ target_config_index < len(scan_config.auto_trade_configs)`; compare current `auto_trade_configs[index]` vs `diff.before` → coerce-or-reject on drift.
  5. **construct the FULL prospective `AutoTradeConfig` (current ⊕ patch)** and run its cross-field validators on THAT.
  6. read-merge-write under TRUE atomic optimistic concurrency → a NEW persistence method `AsyncAnalysisDB.apply_auto_trade_config_atomic(schedule_id, config_index, merged_config, *, expected_prior)` that, in ONE transaction (acquire a dedicated conn, `async with conn.transaction():`): `SELECT scan_config FROM scheduled_scans WHERE id=$1 FOR UPDATE` → **JSONB decode** (`scan_config = row["scan_config"] if isinstance(row["scan_config"], dict) else json.loads(row["scan_config"])` — asyncpg returns JSONB as str, mirror the proven template at `async_persistence.py:1868-1887`, backend-R1-F2) → re-verify `auto_trade_configs[config_index]` still equals `expected_prior` using **canonical comparison** (`AutoTradeConfig(**current).model_dump() == AutoTradeConfig(**expected_prior).model_dump()`, NOT raw dict == — backend-R1-F7) → splice `merged_config` → `UPDATE scheduled_scans SET scan_config=$2, updated_at=now() WHERE id=$1` with **`$2 = json.dumps(merged_scan_config)`**. **(Codebase note C-F24: the existing `update_scheduled_scan` is last-write-wins with no version column and uses `pool.execute` — it CANNOT share a txn with the drift-read, so this new `FOR UPDATE` method is required; do NOT use `update_scheduled_scan` for the apply path. Add `apply_auto_trade_config_atomic` to `_DENY_METHODS` + the call-graph test target — arch-R1-F3.)** Snapshot the prior config (version history); elevated audit record (approver, diff, source sweep_id) via the single audit writer.
- TASK-P4-11: `POST /proposals/{id}/reject`, `/{id}/revert` (restore `diff.before` THROUGH the same sanitize→ceiling→validate pipeline + the atomic apply method; warn if the live config changed since apply).

### Live-trading protection (P4)
- TASK-P4-12: RSS/loop circuit-breaker — cgroup-v2 `memory.current` watchdog sheds sweeps before kernel OOM; live-SLI breaker (event-loop lag p95, scanner cycle, reconciler/order latency, pool-wait) suspends MCP work with hysteresis + flap cap; out-of-band kill path (thread/SIGTERM) works under a saturated loop. `max_connections` preflight gate.
- TASK-P4-12c (QA-R1-F1/F2): build `tests/backend/mcp/conftest.py::synthetic_order_reconciler` fixture — drives the REAL order-placement + reconciler code paths against fakes, records per-cycle latency; its idle run captures the pre-enable BASELINE (hermetic, no external SLI dependency). The live-SLI export (A-005) needed by the production breaker is an explicit prerequisite: if absent, the breaker is fail-closed (suspends MCP) and enable preflight refuses — TASK-P4-12 owns wiring the SLIs the breaker reads, stubbing only what the synthetic fixture provides in tests.
